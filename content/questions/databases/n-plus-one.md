Что такое проблема N+1? Покажите, как она возникает с ORM, способы решения и их компромиссы. Почему GraphQL особенно ей подвержен?

<!-- answer -->

**N+1** — один запрос за списком + по запросу на каждую связь каждого элемента:

```ts
const posts = await prisma.post.findMany();          // 1 запрос
for (const post of posts) {
  const author = await prisma.user.findUnique({      // N запросов
    where: { id: post.authorId },
  });
}
```

100 постов = 101 запрос. Каждый — сетевой roundtrip к БД (~1 мс+) — латентность растёт линейно, пул соединений забивается. Коварство: на дев-данных (5 строк) незаметно.

**Решения:**

1. **Eager loading** — декларировать связь, ORM сделает JOIN или два запроса c `IN`:

```ts
prisma.post.findMany({ include: { author: true } });
// SELECT * FROM posts; SELECT * FROM users WHERE id IN (...)
```

2. **Ручной батчинг:** собрать id, один `WHERE id IN (...)`, разложить по Map.
3. **DataLoader-паттерн** — автоматический батчинг: вызовы `load(id)` в пределах одного тика собираются в пачку + кэш по ключу на время запроса. Стандарт для **GraphQL**, где N+1 встроен в архитектуру: резолвер `Post.author` вызывается независимо для каждого поста, и никакой резолвер не знает о соседях — только DataLoader склеивает их запросы.
4. **JOIN одним запросом / выборка ровно нужных полей** (`select`), агрегаты на стороне БД вместо загрузки строк.

**Компромиссы:** JOIN с one-to-many умножает строки (декартово разбухание при нескольких коллекциях — иногда два запроса дешевле); eager всего подряд тащит лишние данные (over-fetching); DataLoader-кэш живёт в пределах запроса — не путать с межзапросным кэшем.

**Обнаружение:** логирование запросов с подсчётом на HTTP-запрос (алерт на > порога), APM/tracing (span'ы БД повторяющейся формы), Prisma/TypeORM логи в dev, `pg_stat_statements` — топ по calls с одинаковой нормализованной формой.
