Расшифруйте ACID. Какие аномалии конкурентного доступа существуют и какие уровни изоляции их устраняют? Как на практике предотвратить lost update при «прочитал — изменил — записал»?

<!-- answer -->

**ACID:** Atomicity — всё или ничего (журнал/WAL, откат); Consistency — из валидного состояния в валидное (ограничения, триггеры); Isolation — конкурентные транзакции не видят промежуточных состояний друг друга (степень — уровень изоляции); Durability — закоммиченное переживает сбой (fsync WAL).

**Аномалии → уровни:**

| Аномалия | Read Committed | Repeatable Read | Serializable |
|---|---|---|---|
| Dirty read (чужой некоммит) | нет | нет | нет |
| Non-repeatable read (перечитал — другое) | возможна | нет | нет |
| Phantom read (новые строки в повторном запросе) | возможна | нет (в PG) | нет |
| Write skew / аномалии сериализации | возможна | возможна | **нет** |

PostgreSQL по умолчанию — **Read Committed**: каждый **statement** видит свежий снапшот. **Repeatable Read** — один снапшот на всю транзакцию (MVCC), при конфликте записи — ошибка сериализации (нужен retry). **Serializable** (SSI) ловит и write skew — паттерны, где две транзакции читают пересекающиеся данные и пишут в разные места, нарушая инвариант (классика: два врача одновременно снимают дежурство, проверив «остаётся ≥1»).

**MVCC:** запись создаёт новую версию строки, читатели видят версию своего снапшота — **читатели и писатели не блокируют друг друга**. Цена — устаревшие версии и VACUUM.

**Lost update** (два процесса читают баланс 100, оба пишут 100−10 → потеряли одно списание). Решения:

```sql
-- 1. Атомарный апдейт (лучшее, когда возможно)
UPDATE accounts SET balance = balance - 10 WHERE id = 1;

-- 2. Пессимистичная блокировка
SELECT balance FROM accounts WHERE id = 1 FOR UPDATE;  -- держит до commit

-- 3. Оптимистичная (версия)
UPDATE accounts SET balance = ?, version = version + 1
WHERE id = 1 AND version = ?;  -- 0 строк → конфликт, повторить
```

Плюс Repeatable Read/Serializable с ретраями. Выбор: горячие строки — пессимистично коротко; редкие конфликты/долгие формы — оптимистично. Помнить: долгие транзакции держат снапшоты и блокировки — держать их короткими, внешние вызовы вне транзакции.
