Приложение деплоится без даунтайма (rolling deploy): старый и новый код работают одновременно. Какие правила накладывает это на миграции БД? Разберите переименование колонки и добавление NOT NULL.

<!-- answer -->

**Ключевой факт:** во время rolling-деплоя (и при откате) **старый код работает с новой схемой**. Отсюда правило: каждая миграция обратно совместима с предыдущей версией кода — схема и код деплоятся **раздельными шагами** (expand → migrate → contract).

**Переименование колонки `name` → `full_name`** (одним `RENAME` — старый код мгновенно падает):

1. **Expand:** добавить `full_name` (nullable); код пишет **в обе**, читает старую.
2. **Backfill:** скопировать данные батчами (не одним UPDATE — блокировки, WAL, bloat).
3. Переключить чтение на новую; писать всё ещё в обе (безопасен откат).
4. Прекратить запись в старую.
5. **Contract:** удалить старую колонку (отдельный деплой спустя время).

То же для смены типа, переноса в другую таблицу, разделения таблиц. Скучно, зато без 3 часов ночного окна.

**Опасные операции (блокирующие таблицу):**

- `ADD COLUMN NOT NULL` без default на непустой таблице — нельзя; с `DEFAULT`: в PG 11+ это метаданные (быстро), в старых — переписывание таблицы. Безопасный путь: nullable → backfill → `SET NOT NULL` (в PG: сначала `ADD CONSTRAINT ... CHECK (col IS NOT NULL) NOT VALID` → `VALIDATE CONSTRAINT` (без долгой блокировки) → `SET NOT NULL`).
- **Индекс**: `CREATE INDEX` блокирует запись → всегда `CREATE INDEX CONCURRENTLY` (вне транзакции, уметь чинить invalid-индексы при сбое).
- Внешний ключ: `NOT VALID` → `VALIDATE` отдельно.
- Любой `ALTER`, ждущий ACCESS EXCLUSIVE, встаёт **в очередь за долгими запросами** и собирает за собой всех: ставить `lock_timeout` (2–5 с) + ретраи.

**Процесс:** миграции в git и CI (линтер опасных операций), forward-only с планом отката «накатить следующую», данные-миграции отдельно от схемных, тестировать на копии прод-объёма. ORM-автомиграции (`prisma migrate`) генерируют логически корректный, но не всегда безопасный SQL — на больших таблицах проверять руками.
