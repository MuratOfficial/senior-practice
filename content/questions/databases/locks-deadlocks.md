Какие блокировки берёт Postgres при записи? Разберите deadlock: как возникает, как обнаруживается, как проектировать код без дедлоков. Что такое SELECT FOR UPDATE, SKIP LOCKED и advisory locks?

<!-- answer -->

**База: MVCC + блокировки записи.** Чтения в Postgres не блокируют и не блокируются (MVCC — читатель видит снапшот). Блокировки — про **конкурентную запись**: `UPDATE/DELETE` берёт row-level lock на изменяемые строки до конца транзакции; второй писатель той же строки **ждёт** коммита первого. Плюс table-level locks (DDL против DML — почему `ALTER TABLE` может встать в очередь и собрать за собой всех).

**Deadlock:** T1 залочила строку A и ждёт B; T2 залочила B и ждёт A. Postgres периодически (deadlock_timeout, 1s) строит граф ожиданий, находит цикл и **убивает одну транзакцию** (deadlock detected) — приложение должно уметь ретраить.

**Профилактика — упорядочивание:** брать блокировки **в одинаковом порядке** во всех код-путях: батчи `UPDATE ... WHERE id IN (...)` — предварительно отсортировать id; перевод денег между счетами — лочить счета по возрастанию id, а не «сначала from, потом to». Плюс: короткие транзакции (не держать lock на время внешнего HTTP-вызова!), меньше строк под локом, ретрай с backoff как страховка.

**`SELECT ... FOR UPDATE`** — явно залочить строки при чтении: паттерн read-modify-write без гонки (прочитать баланс → изменить → записать в одной транзакции). Варианты: `FOR NO KEY UPDATE` (мягче — не конфликтует со вставкой FK-ссылок), `FOR SHARE` (разделяемая). Модификаторы ожидания: `NOWAIT` (ошибка сразу), **`SKIP LOCKED`** — пропустить занятые строки: канонический паттерн **очереди задач в Postgres** (`SELECT ... WHERE status='pending' ORDER BY id LIMIT 10 FOR UPDATE SKIP LOCKED`) — воркеры разбирают задачи без конкуренции.

**Advisory locks** — блокировки уровня приложения по произвольному ключу (`pg_advisory_lock(key)` / сессионные и транзакционные `pg_advisory_xact_lock`): распределённый мьютекс без строки-носителя — «одна миграция за раз», «один cron-инстанс», сериализация обработки сущности.

**Долгие транзакции вредны шире блокировок:** держат xmin → мешают VACUUM убирать мёртвые версии → раздувание таблиц; удерживают локи; растят риск конфликтов сериализации. Мониторинг: `pg_locks` + `pg_stat_activity` (`wait_event`), `log_lock_waits = on`.

**Альтернатива пессимизму** — оптимистическая блокировка: колонка `version`, `UPDATE ... WHERE id=$1 AND version=$2`; 0 строк → конфликт, повторить. Выбор: высокая конкуренция за строку — пессимистичная; редкие конфликты — оптимистичная (без ожиданий и дедлоков).
