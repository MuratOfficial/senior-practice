Как устроен B-tree индекс и почему поиск по нему быстрый? Объясните составные и покрывающие индексы, правило левого префикса. В каких случаях существующий индекс не будет использован?

<!-- answer -->

**B-tree** — сбалансированное дерево с большим фактором ветвления: узел — страница (~8 КБ) с сотнями ключей; высота дерева для миллионов строк — 3–4 уровня → поиск за 3–4 чтения страниц, O(log n). Листья содержат ключи + ссылки на строки (и связаны списком → эффективные range-сканы и ORDER BY по индексу). Поддерживает `=`, `<`, `>`, `BETWEEN`, `LIKE 'prefix%'`.

**Составной индекс `(a, b, c)`** отсортирован лексикографически: сначала a, внутри — b, внутри — c. Отсюда **правило левого префикса**: работают `WHERE a=?`, `a=? AND b=?`, `a=? AND b=? AND c>?`; не работают `WHERE b=?` (b разбросан по всему дереву). Диапазон «останавливает» использование следующих колонок: при `a>? AND b=?` по b будет фильтрация, не поиск. Порядок колонок: равенства слева, диапазоны/сортировки правее.

**Покрывающий индекс** — содержит все колонки запроса: `SELECT b FROM t WHERE a=?` с индексом `(a, b)` (или `(a) INCLUDE (b)`) не обращается к таблице вообще (Index-Only Scan). Убирает главную скрытую стоимость — случайные чтения строк из heap.

**Когда индекс не используется:**

- **функция/приведение над колонкой**: `WHERE lower(email)=?`, `WHERE date(created_at)=?`, сравнение varchar с числом — нужен функциональный индекс или переписать условие диапазоном;
- **низкая селективность**: условие выбирает большой процент таблицы — full scan дешевле тысяч случайных чтений; планировщик прав (индекс по boolean почти бесполезен, но частичный индекс `WHERE status='pending'` по редкому значению — работает);
- `LIKE '%substr'` (нет префикса), `OR` по разным колонкам (иногда спасает BitmapOr), ведущий wildcard, `!=`;
- устаревшая статистика (`ANALYZE`), несоответствие collation/типов в JOIN.

**Цена:** каждый индекс — доп. запись при INSERT/UPDATE/DELETE, место, замедление вакуума; невыгодные индексы искать через `pg_stat_user_indexes` (idx_scan = 0). Диагностика запросов — `EXPLAIN (ANALYZE, BUFFERS)`: смотреть реальные строки против оценки и Buffers.

Другие типы: Hash (только `=`), GIN (массивы, jsonb, полнотекст), GiST/BRIN (гео, большие «естественно отсортированные» таблицы).
