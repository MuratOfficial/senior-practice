Как реляционная БД решает, каким способом выполнить запрос? Объясните роль планировщика и статистики, как читать EXPLAIN/EXPLAIN ANALYZE, когда seq scan предпочтительнее индекса и как диагностировать плохой план.

<!-- answer -->

**Планировщик (optimizer).** SQL декларативен — вы говорите *что* нужно, а не *как*. БД рассматривает разные физические планы (порядок джойнов, методы доступа, алгоритмы соединения) и по **оценке стоимости** выбирает самый дешёвый. Стоимость считается по **статистике**: число строк в таблице, кардинальность (уникальность) столбцов, гистограммы распределения, доля NULL. В Postgres статистику собирает `ANALYZE` (обычно autovacuum).

**EXPLAIN vs EXPLAIN ANALYZE:**

- `EXPLAIN` — показывает **предполагаемый** план и оценки (`cost`, `rows`), запрос **не выполняя**.
- `EXPLAIN ANALYZE` — **реально выполняет** запрос и показывает фактическое время и строки. Опасность: для `INSERT/UPDATE/DELETE` он **реально изменит данные** — оборачивайте в транзакцию с `ROLLBACK` (или используйте `BEGIN; ... ROLLBACK;`).

Читать план — **снизу вверх и изнутри наружу**: листья (сканы таблиц) → джойны → сортировка/агрегация наверху. Ключевое — сравнить **estimated rows vs actual rows**: большое расхождение (напр. план ждёт 10 строк, а пришло 100000) — сигнал неверной статистики или коррелированных условий, из-за чего выбран плохой план.

```
Nested Loop  (rows=10) (actual rows=98000 ...)   ← оценка мимо на 4 порядка
  -> Seq Scan on orders (actual rows=98000)
  -> Index Scan on users ...
```

Здесь планировщик выбрал Nested Loop, ожидая мало строк, а по факту гоняет индекс 98000 раз — надо было Hash Join.

**Когда seq scan быстрее индекса.** Индекс не всегда выигрывает:

- Запрос возвращает **большую долю** таблицы (напр. >5–20%): по индексу пришлось бы прыгать за каждой строкой в heap (random I/O), а последовательное чтение всей таблицы (sequential I/O) дешевле.
- Таблица маленькая — целиком в кэше, индекс избыточен.
- Условие **не селективно** (`WHERE active = true`, где 90% активны).

Планировщик выбирает seq scan осознанно — и часто прав. Тревога — когда он делает seq scan на **селективном** условии по большой таблице: тогда причина обычно в отсутствии индекса, устаревшей статистике, или несовместимости условия с индексом (функция над столбцом, неявный каст, `LIKE '%x'`).

**Плохой план из-за статистики.** После массовой загрузки/удаления статистика устаревает — оценки кардинальности врут, план деградирует. Лечение: `ANALYZE table` (обновить статистику), поднять `default_statistics_target` для «перекошенных» столбцов, `CREATE STATISTICS` для **коррелированных** колонок (планировщик по умолчанию считает условия независимыми и перемножает селективности — при корреляции сильно ошибается).

**Метод диагностики:** воспроизвести медленный запрос → `EXPLAIN (ANALYZE, BUFFERS)` → найти узел с расхождением estimate/actual и большим временем → проверить индексы/статистику/селективность → при необходимости переписать запрос (убрать функцию со столбца, добавить составной/покрывающий индекс) и сравнить план снова.
