Что такое GIL и зачем он существует? Как выбрать между threading, multiprocessing и asyncio для конкретной задачи? Приведите правило выбора.

<!-- answer -->

**GIL (Global Interpreter Lock)** — мьютекс CPython, позволяющий только одному потоку исполнять байткод в момент времени. Существует, потому что управление памятью CPython основано на **подсчёте ссылок** без атомарности — GIL защищает счётчики и внутренние структуры дёшево и просто (однопоточный код быстрее, C-расширения проще).

**Следствия:**

- **CPU-bound**: потоки не дают параллелизма — два потока считают на одном ядре по очереди (даже медленнее из-за борьбы за GIL).
- **I/O-bound**: потоки работают — при блокирующем I/O (сеть, диск, sleep) поток **освобождает GIL**, другие исполняются. То же делают многие C-расширения (numpy на больших массивах, hashlib) — освобождают GIL внутри тяжёлых секций.

**Правило выбора:**

| Задача | Инструмент |
|---|---|
| CPU-bound (парсинг, вычисления) | `multiprocessing` / `ProcessPoolExecutor` — отдельные процессы, свой GIL у каждого |
| I/O-bound, десятки задач | `threading` / `ThreadPoolExecutor` — просто, совместимо с блокирующими библиотеками |
| I/O-bound, тысячи соединений | `asyncio` — кооперативная многозадачность в одном потоке, дёшево по памяти |

**Цена процессов:** данные между ними **сериализуются** (pickle) — передача больших объектов может съесть весь выигрыш; память не разделяется (форк копирует copy-on-write на Unix, spawn на Windows — полный старт интерпретатора); межпроцессная синхронизация дороже. Разделяемая память — `multiprocessing.shared_memory`.

**Динамика:** Python 3.13+ имеет экспериментальную **free-threaded** сборку (PEP 703, GIL отключаем): честный параллелизм потоков ценой замедления однопоточного кода и требований к потокобезопасности C-расширений; также `interpreters` (PEP 734) — субинтерпретаторы с GIL на каждый. На интервью важно показать: GIL — свойство CPython, а не языка (Jython/GraalPy без него), и его снятие — вопрос экосистемы расширений.
