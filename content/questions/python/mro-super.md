Что такое MRO и как строится C3-линеаризация? Что на самом деле делает `super()` при множественном наследовании? Почему «diamond»-иерархия работает корректно только при кооперативном вызове super во всех классах?

<!-- answer -->

**MRO (Method Resolution Order)** — порядок, в котором Python ищет атрибут/метод по классам-предкам. Для множественного наследования он вычисляется алгоритмом **C3-линеаризации**: строит единый линейный список так, что (1) сохраняется порядок базовых классов из объявления, (2) каждый класс идёт **перед** своими родителями, (3) результат монотонен. Смотреть — `Cls.__mro__` или `Cls.mro()`.

**`super()` — это не «родитель», а «следующий в MRO».** Ключевое заблуждение: `super().method()` не вызывает метод суперкласса — он вызывает следующий класс **в MRO текущего экземпляра**, а MRO зависит от реального типа объекта, а не от того, где написан `super`. Поэтому в diamond-иерархии `super()` из класса B может уйти в C (сиблинг), а не в общий предок:

```python
class A:
    def __init__(self): print("A"); # общий предок
class B(A):
    def __init__(self): print("B"); super().__init__()
class C(A):
    def __init__(self): print("C"); super().__init__()
class D(B, C):
    def __init__(self): print("D"); super().__init__()

D()                 # D, B, C, A  — каждый ровно один раз
D.__mro__           # D -> B -> C -> A -> object
```

Из `B.__init__` `super()` для экземпляра `D` идёт **в C**, не в A — потому что в MRO(D) после B стоит C. Так «ромб» инициализирует общий предок A **один раз**, а не дважды.

**Почему нужен кооператив.** Это работает только если **каждый** класс в цепочке зовёт `super().__init__()` (и передаёт совместимые аргументы, обычно через `*args, **kwargs`). Стоит одному классу вызвать напрямую `A.__init__(self)` или забыть `super()` — цепочка рвётся: часть предков не инициализируется или инициализируется дважды. Отсюда паттерн «кооперативное множественное наследование»: методы проектируют так, чтобы их можно было ставить в цепочку в любом порядке.

**Когда C3 невозможен.** Если требования порядка **противоречивы** — например `class D(B, C)` при иерархии, где B требует X перед Y, а C требует Y перед X, — согласованной линеаризации нет, и Python бросает `TypeError: Cannot create a consistent method resolution order (MRO)` прямо при определении класса. Это защита от неоднозначности, а не рантайм-ошибка.

**Практика:** множественное наследование в Python оправдано для **миксинов** (узкие, ортогональные добавки поведения), где кооперативный `super` естественен. Для сложных «ромбов» с состоянием — часто чище композиция. Всегда проверяйте `__mro__`, когда поведение при наследовании удивляет.
