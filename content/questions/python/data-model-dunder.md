Как работает протокольная модель Python через dunder-методы? Объясните контракт `__eq__`/`__hash__`, зачем и когда нужен `__slots__`, и разницу между `__getattr__` и `__getattribute__`.

<!-- answer -->

**Data model.** В Python поведение объектов задаётся **специальными методами** (`__len__`, `__iter__`, `__getitem__`, `__enter__`, `__add__`…). `len(x)` → `x.__len__()`, `x + y` → `x.__add__(y)`, `for i in x` → `x.__iter__()`. Это протоколы: реализуешь нужные dunder'ы — объект «утиный» для соответствующих операций, наследование не требуется.

**Контракт `__eq__`/`__hash__`.** Два объекта, равные по `__eq__`, **обязаны** иметь одинаковый `__hash__` — иначе они сломаются как ключи `dict`/элементы `set` (объект не найдётся в своей же корзине). Ловушка: как только вы определяете `__eq__`, Python **сбрасывает `__hash__` в `None`** — объект становится нехэшируемым:

```python
class Point:
    def __init__(self, x, y): self.x, self.y = x, y
    def __eq__(self, other):
        return isinstance(other, Point) and (self.x, self.y) == (other.x, other.y)
    # без __hash__ — TypeError: unhashable type в set/dict

    def __hash__(self):
        return hash((self.x, self.y))  # согласован с __eq__
```

Правила: хэш считаем из тех же полей, что и равенство; хэшировать стоит только **иммутабельные** объекты (изменив поле-ключ, вы «потеряете» объект в set). `@dataclass(frozen=True)` делает это правильно автоматически; `@dataclass(eq=True)` без frozen — оставляет объект нехэшируемым осознанно.

**`__slots__`** отменяет пер-инстансный `__dict__`: атрибуты хранятся в фиксированном массиве, а не в словаре.

```python
class Point:
    __slots__ = ("x", "y")  # нет __dict__
```

Даёт: заметную **экономию памяти** (критично для миллионов мелких объектов) и небольшое ускорение доступа. Цена: нельзя добавлять произвольные атрибуты вне списка (`p.z = 1` → `AttributeError`), сложности с множественным наследованием (конфликт slots), нет `__weakref__` без явного добавления, ломается наивный `__dict__`-based код. Применять точечно — к «структурам данных», а не ко всему подряд.

**`__getattr__` vs `__getattribute__`:**

- `__getattribute__(self, name)` вызывается на **каждый** доступ к атрибуту `obj.name` — перехватывает всё. Переопределять опасно (легко словить бесконечную рекурсию; внутри обращайтесь через `super().__getattribute__`).
- `__getattr__(self, name)` — **фолбэк**, срабатывает только когда обычный поиск **не нашёл** атрибут. Идеален для прокси, ленивых полей, ORM-магии, обёрток:

```python
class Config:
    def __getattr__(self, name):   # только для отсутствующих
        return self._data.get(name)
```

По умолчанию для расширения атрибутного доступа берут `__getattr__` (безопасно), а `__getattribute__` — только когда нужно вмешаться абсолютно во всё.
