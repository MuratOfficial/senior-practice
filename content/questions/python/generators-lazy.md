Объясните протокол итерации в Python и генераторы. Чем генераторное выражение отличается от list comprehension по памяти и семантике? Когда ленивость — правильный выбор?

<!-- answer -->

**Протокол:** `iter(obj)` вызывает `__iter__` → итератор с `__next__`, который выдаёт значения и бросает `StopIteration` в конце. `for` делает это неявно. Итератор одноразов; iterable может выдавать новые итераторы.

**Генератор** — функция с `yield`: вызов возвращает генератор-объект (итератор), тело исполняется лениво между yield'ами с сохранением локального состояния. Это самый дешёвый способ написать итератор без класса.

```python
def read_large(path):
    with open(path) as f:
        for line in f:          # файл не грузится целиком
            yield parse(line)

total = sum(r.amount for r in read_large("big.csv"))  # O(1) память
```

**Genexpr vs list comprehension:** `[x*x for x in data]` строит весь список в памяти сразу; `(x*x for x in data)` — ленивый генератор, элементы по требованию. Отличия семантики: генератор **одноразовый** (второй проход пуст — классический баг), не имеет `len`, не индексируется, ошибки всплывают при потреблении, а не при создании. Ленивость по цепочке даёт **конвейеры**: `filtered = (f(x) for x in источник)` — ни одного промежуточного списка, O(1) память на любой длине потока (модель Unix-pipe; `itertools` — готовые комбинаторы: islice, chain, groupby).

**Когда ленивость правильна:** большие/бесконечные последовательности, стриминг (файлы, БД-курсоры, API-пагинация), ранний выход (`next(x for x in items if pred(x))` — не считает лишнего). Когда нет: нужны множественные проходы, len, случайный доступ, или важно немедленно провалидировать данные.

**Продвинутое:** `send(value)` превращает генератор в корутину (значение становится результатом yield); `throw`/`close` — инъекция исключений (на этом строились асинхронные фреймворки до async/await); **`yield from gen`** — делегирование: прозрачно пробрасывает значения, send и исключения во вложенный генератор и возвращает его `return`-значение.
