Как React решает, что обновить в DOM? Опишите алгоритм согласования (reconciliation), эвристики диффа и роль `key`. Какие баги возникают при неправильных key?

<!-- answer -->

Рендер в React — две фазы: **render** (вызов компонентов, построение нового дерева React-элементов — чистая, прерываемая) и **commit** (применение минимальных изменений к DOM). Сравнение старого и нового деревьев — **reconciliation**.

Честный дифф деревьев — O(n³), поэтому React использует **две эвристики**:

1. **Разные типы — разное поддерево.** `<div>` → `<span>` или `CompA` → `CompB`: старое поддерево размонтируется полностью (state теряется), новое монтируется с нуля. Одинаковый тип — обновляются только пропсы, state сохраняется.
2. **key идентифицирует детей в списке.** Без key сравнение по позиции; с key React сопоставляет элементы между рендерами независимо от позиции.

**Баги неправильных key:**

- `key={index}` + вставка/удаление/сортировка: state «прилипает» к позиции, а не к данным — введённый текст в инпуте оказывается у чужого элемента, анимации дёргаются, uncontrolled-поля путаются;
- `key={Math.random()}` — каждый рендер полное перемонтирование: потеря state, фокуса, производительности;
- дубликаты key — непредсказуемое поведение.

Правильный key — **стабильный идентификатор данных** (id из БД). Index допустим, если список статичен: не сортируется, не фильтруется, элементы без состояния.

**key как инструмент:** смена key — легальный способ принудительно сбросить state компонента (`<Profile key={userId} />` — новый пользователь, чистая форма).

**Fiber** — архитектура reconciler'а (React 16+): дерево fiber-узлов со ссылками child/sibling/return, работа разбита на прерываемые единицы. Это фундамент конкурентного рендера: приоритеты обновлений, прерывание и возобновление render-фазы (`startTransition`), Suspense. Важно следствие: render-фаза может выполняться **несколько раз** — отсюда требование чистоты рендера и опасность side effects в теле компонента.
