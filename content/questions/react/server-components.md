Объясните модель React Server Components: чем серверный компонент отличается от клиентского, как они комбинируются, что такое граница "use client". Как RSC соотносится с классическим SSR?

<!-- answer -->

**Две разные оси.** SSR — **где выполняется первый рендер** клиентского кода (HTML на сервере → гидрация). RSC — **какие компоненты вообще существуют только на сервере**: их код не попадает в бандл, результат сериализуется в специальный формат (RSC payload) и передаётся клиенту как готовое дерево.

**Серверные компоненты** (по умолчанию в Next.js App Router):

- могут быть async, напрямую читать БД/файлы/секреты;
- ноль JS в клиентском бандле; библиотека для markdown на 200 КБ остаётся на сервере;
- не имеют state, эффектов, обработчиков событий, browser API.

**Клиентские** (`"use client"` в начале файла) — обычный React: интерактивность, хуки, браузерные API. Директива помечает **границу**: всё импортируемое этим модулем тоже попадает в клиентский бандл.

**Комбинирование:** серверные могут рендерить клиентские и передавать им **сериализуемые** пропсы (не функции, кроме server actions). Клиентский не может импортировать серверный, но может принять его **как children** — паттерн «серверный контент внутри интерактивной обёртки»:

```tsx
<ClientTabs>            {/* "use client" */}
  <ServerArticle />     {/* отрендерен на сервере, передан как children */}
</ClientTabs>
```

**Гидрация** — привязка обработчиков и построение fiber-дерева поверх серверного HTML. Клиентские компоненты при SSR рендерятся на сервере в HTML, затем гидрируются. Mismatch (dates, random, browser-only ветки) — ошибка: разметка сервера и первый клиентский рендер обязаны совпадать (`suppressHydrationWarning` точечно, `useEffect` для browser-only значений).

**Стратегии по типу страницы:** статичный маркетинг/доки — **SSG**; каталог с редкими изменениями — **ISR** (регенерация по времени/`revalidateTag`); персонализированное/актуальное — **SSR** (+ streaming с Suspense: оболочка сразу, данные по мере готовности); дашборды за логином с интенсивным взаимодействием — CSR поверх серверной оболочки. Решение — per-страница, а не per-приложение.
