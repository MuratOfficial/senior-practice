Как вы проектируете состояние в крупном React-приложении? Классифицируйте виды состояния, объясните ограничения Context API и критерии выбора внешнего стора.

<!-- answer -->

**Сначала классификация — она определяет инструмент:**

1. **Локальное UI-состояние** (открыт ли дропдаун, значение инпута) — `useState`/`useReducer` в компоненте. Поднимать выше только при реальной необходимости.
2. **Серверное состояние** (данные API) — это **кэш чужих данных** со своими проблемами: устаревание, инвалидация, ретраи, дедупликация, optimistic updates. TanStack Query / SWR / RSC. Класть его в Redux — значит вручную писать плохой кэш.
3. **Глобальное клиентское** (тема, корзина, текущий пользователь) — Context для редко меняющегося; Zustand/Redux/Jotai для часто меняющегося и сложного.
4. **Состояние в URL** (фильтры, пагинация, вкладка) — searchParams: шарится ссылкой, переживает refresh.
5. **Формы** — react-hook-form / server actions.

**Context — механизм внедрения (DI), а не стейт-менеджер:** при изменении value ререндерятся **все** потребители, селекторов нет. Объект-value без useMemo — новая ссылка каждый рендер родителя → каскад. Паттерны смягчения: разделить контексты (state и dispatch отдельно), мемоизировать value, выносить быстро меняющееся из контекста.

**Внешние сторы** (Zustand/Redux) подписывают компоненты **точечно через селекторы** — ререндер только у тех, чей срез изменился. Интеграция с React — `useSyncExternalStore`: корректная подписка + защита от **tearing** (в конкурентном рендере разные части дерева могли бы увидеть разные версии стора).

```ts
const useCart = create((set) => ({
  items: [],
  add: (item) => set((s) => ({ items: [...s.items, item] })),
}));
const count = useCart((s) => s.items.length); // подписка на срез
```

**Критерии выбора:** начинать с локального состояния + серверный кэш + URL; глобальный стор вводить, когда есть реально разделяемое клиентское состояние с нетривиальной логикой. Redux Toolkit — когда нужны строгие паттерны, devtools time-travel, большая команда; Zustand — минимум церемоний; Jotai — атомарная модель для сильно связанных мелких кусочков.
