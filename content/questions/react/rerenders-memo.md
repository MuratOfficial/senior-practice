Когда компонент ререндерится? Объясните, как реально работают `React.memo`, `useMemo`, `useCallback`, когда они помогают, когда бесполезны и когда вредны. Какие есть способы убрать ререндеры без мемоизации?

<!-- answer -->

**Компонент ререндерится, когда:** изменился его state; заререндерился родитель (по умолчанию — всё поддерево); изменился используемый контекст. Изменение пропсов само по себе не триггер — пропсы «меняются» потому, что рендерится родитель.

**Инструменты:**

- `React.memo(Comp)` — пропускает рендер, если пропсы поверхностно равны (`Object.is` по каждому).
- `useMemo(fn, deps)` — кэширует **значение** между рендерами.
- `useCallback(fn, deps)` — кэширует **ссылку на функцию** (= `useMemo(() => fn, deps)`).

**Почему memo часто не работает:** инлайн-объекты, массивы и функции пересоздаются каждый рендер — новая ссылка, сравнение падает. `children` (JSX-элемент) — тоже новый объект каждый рендер. Поэтому memo-компонент требует стабильных ссылок во **всех** пропсах: useMemo/useCallback у родителя. Одна забытая — вся мемоизация мертва.

**Когда оправдано:** тяжёлый рендер (большие списки, графики), дорогие вычисления, стабильность deps для чужих useEffect. **Когда вредно:** мемоизация всего подряд — накладные расходы на сравнение и память, замусоренный код, ложное чувство оптимизации без профилирования (React DevTools Profiler — сначала измерить).

**Без мемоизации (композиция):**

```tsx
// Родитель со state не обязан рендерить тяжёлое поддерево:
function Page() {
  return <ScrollTracker content={<HeavyTree />} />; // или children
}
```

Состояние живёт в `ScrollTracker`, а `content` — элемент, созданный **вне** него: при его ререндерах проп не менялся → React переиспользует поддерево. Плюс «опускание state вниз» — держать состояние максимально близко к месту использования.

**React Compiler** автоматически мемоизирует компоненты и значения на этапе сборки — ручные useMemo/useCallback становятся в основном ненужными, но понимание модели ререндеров остаётся обязательным (компилятор не спасает от неправильной архитектуры состояния и нестабильных внешних данных).
