Почему хуки нельзя вызывать в условиях и циклах? Как React хранит состояние хуков между рендерами? Что выведет код при трёх кликах подряд и почему?

```jsx
function Counter() {
  const [n, setN] = useState(0);
  const onClick = () => {
    setN(n + 1);
    setN(n + 1);
  };
  // ...
}
```

<!-- answer -->

**Хранение:** у каждого fiber-узла есть **связный список ячеек хуков**. При рендере React выдаёт ячейки строго **по порядку вызова**: первый useState — первая ячейка, второй — вторая. Имён у хуков нет — только позиция. Условный вызов сдвигает порядок: третий хук получает состояние четвёртого — тихая порча данных. Отсюда правила: только на верхнем уровне компонента/кастомного хука, порядок стабилен между рендерами (проверяет eslint-plugin-react-hooks).

**Пример: после трёх кликов n = 3, не 6.** В момент клика `n` — значение из замыкания текущего рендера (например, 0). Оба вызова: `setN(0 + 1)`. React батчит обновления и берёт последнее одинаковое значение. Функциональный апдейтер работает с актуальным значением из очереди:

```js
setN(prev => prev + 1);
setN(prev => prev + 1); // теперь +2 за клик
```

Правило: **новое состояние зависит от старого → всегда updater-функция.**

**Модель снапшотов:** каждый рендер — «фотография»: пропсы, state и все созданные функции видят значения своего рендера. **Stale closure** — колбэк из старого рендера (в setInterval, подписке, промисе) использует устаревший state. Лечится: updater-функции, корректные зависимости эффекта, useRef для «всегда актуального» значения.

**Батчинг:** React 18+ группирует setState отовсюду (события, промисы, таймеры) — один ререндер на серию обновлений; `flushSync` — принудительный синхронный сброс (редко, для измерения DOM).

**useState vs useRef:** ref — та же «ячейка на fiber», но изменение `.current` не планирует рендер и доступно синхронно. Всё, что не влияет на разметку (id таймера, предыдущее значение, флаги), — в ref; в state — только то, что рисуется.
