Спроектируйте мессенджер: диалоги 1:1 и группы, доставка в реальном времени, офлайн-доставка, несколько устройств. Разберите транспорт, маршрутизацию сообщений между серверами соединений и хранение истории.

<!-- answer -->

**1. Транспорт.** Постоянное соединение — **WebSocket** (двунаправленность; fallback — SSE/long polling). Клиент держит одно соединение с **connection-сервером**; серверов много, пользователь может попасть на любой → центральная задача: **маршрутизация** «на каком сервере сидит получатель».

**2. Маршрутизация.** Реестр присутствия: `userId → connection server` в Redis (ставится при подключении, TTL + heartbeat). Отправка: сообщение → сервис сообщений → находит сервер получателя → доставляет через внутренний pub/sub (Redis pub/sub, NATS, Kafka по каналу сервера). Получатель офлайн → сообщение только в хранилище + push-уведомление (APNs/FCM); при подключении клиент догоняет историю (sync по курсору).

**3. Хранение.** Сообщения — append-heavy, чтение «последние N по диалогу»: wide-column (Cassandra/Scylla, партиция = диалог, кластеризация по времени/id) либо шардированный Postgres по conversation_id. Discord: партиции диалогов + бакеты по времени, чтобы партиции не разрастались. Метаданные (участники, настройки) — обычная реляционная БД.

**4. Порядок и id.** Глобальный порядок не нужен — нужен **порядок в пределах диалога**: монотонный id на диалог (последовательность на партиции / snowflake + упорядочивание на сервере диалога). Клиент сортирует по id сервера, не по локальному времени. Дубли при ретраях → **клиентский idempotency key** (uuid сообщения), сервер дедуплицирует.

**5. Группы** — fan-out при отправке: сообщение пишется один раз в диалог, доставка — каждому онлайн-участнику через его connection-сервер. Большие каналы (100k) — не слать всем мгновенно: подписчики читают по курсору, realtime-доставка только активным.

**6. Статусы.** «Отправлено» — принято сервером (ack с id); «доставлено» — ack устройства получателя; «прочитано» — read-курсор диалога (одна отметка «прочитано до id», не пометка на каждом сообщении). Мультиустройство: каждое устройство имеет свой sync-курсор; отправленное с одного устройства доставляется и остальным устройствам автора.

**7. Присутствие (online/last seen).** Heartbeat раз в N сек в реестр с TTL; рассылать изменения статуса не всем контактам сразу, а лениво (при открытии списка) или через подписку на «нужных» пользователей — иначе fan-out статусов превышает трафик сообщений.

**Ключевые trade-offs:** доставка at-least-once + идемпотентность (никогда «ровно один раз по сети»); eventual для статусов, строгий порядок только внутри диалога; шифрование E2E (Signal protocol) — упомянуть, что оно меняет серверное хранение (сервер видит только шифротекст).
