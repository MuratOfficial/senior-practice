Спроектируйте сервис коротких ссылок: создание короткого кода, редирект, статистика переходов. Оцените нагрузку, спроектируйте хранение и генерацию кодов, объясните, где здесь кэш и что станет узким местом.

<!-- answer -->

**1. Требования и оценка.** Уточнить: соотношение чтение/запись (обычно 100:1 — редиректы доминируют), TTL ссылок, кастомные алиасы, аналитика. Прикидка: 100 млн новых ссылок/мес ≈ 40 RPS записи, 4000 RPS чтения; за 5 лет ~6 млрд записей × ~500 байт ≈ 3 ТБ — влезает в одну реплицируемую БД с шардированием «про запас».

**2. Генерация кода.** Ключевое решение:

- **Счётчик + base62** (0-9a-zA-Z): 62⁷ ≈ 3.5 трлн кодов при длине 7. Монотонный id → base62. Проблема: предсказуемость (перебор чужих ссылок) — решается смешиванием (например, bijective-скремблинг числа) и rate limiting.
- **Случайный код с проверкой уникальности**: непредсказуем, но нужен retry на коллизии (уникальный индекс + повтор при конфликте).
- **Хэш от URL** (md5, первые 7 символов) — коллизии чаще, зато одинаковый URL даёт одинаковый код (дедупликация). 

Распределённая генерация: диапазоны счётчика выдаются инстансам блоками (range allocator в БД/ZooKeeper) — нет единой точки на каждый запрос.

**3. Хранение.** Таблица `links(code PK, url, owner, created_at, expires_at)`. Это идеальный key-value доступ — подходит и Postgres, и DynamoDB/Cassandra при росте. Шардирование по коду (hash-based) тривиально, потому что запросы всегда точечные.

**4. Редирект — горячий путь.** `GET /{code}`: кэш (Redis, LRU по горячим кодам — трафик сильно скошен: 20% ссылок дают 80% переходов) → БД при промахе → 301/302. **302 (или 307)**, если нужна аналитика на каждый переход: 301 браузер закэширует и больше не придёт. Ставить CDN перед сервисом можно только с коротким TTL — иначе не деактивировать ссылку.

**5. Аналитика.** Не писать синхронно в БД на каждый переход (умножает запись ×100): событие перехода → очередь/лог (Kafka) → агрегация батчами (счётчики по часам, гео, referrer). Потеря единичных событий допустима — это метрики, не деньги.

**6. Узкие места и отказы.** Горячая ссылка (вирусный пост) — держит кэш; недоступность БД — редиректы продолжают работать из кэша (graceful degradation); злоупотребления — rate limiting на создание, блэклисты доменов, проверка malware.

**Структура ответа на интервью:** требования → оценки → API → схема данных → генерация кода (главный trade-off) → кэш/редирект → аналитика → масштабирование и отказы. Проговаривать компромиссы вслух — это и оценивается.
