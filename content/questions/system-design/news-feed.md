Спроектируйте ленту: пользователи подписываются друг на друга, пост автора должен появляться в лентах подписчиков. Сравните подходы fan-out on write и fan-out on read, объясните гибридную схему и работу с «знаменитостями».

<!-- answer -->

**1. Суть задачи** — это join «посты всех, на кого я подписан», который нельзя делать на лету при миллионах пользователей. Всё проектирование — про то, **когда** материализовать этот join.

**2. Fan-out on read (pull).** Лента строится при запросе: взять список подписок → последние посты каждого → слить по времени. Запись дешёвая (1 insert), чтение дорогое: N запросов + merge на каждый рефреш. Работает при малом графе; ломается на пользователе с 1000 подписок.

**3. Fan-out on write (push).** При публикации пост (или его id) **раскладывается в ленты всех подписчиков** — precomputed feed в Redis: `feed:{userId}` → список post_id (обрезается до ~800 записей). Чтение — O(1): один запрос к своему списку + догрузка тел постов по id (кэш постов). Запись дорогая: пост автора с 10 тыс. подписчиков = 10 тыс. записей, выполняется **асинхронно** через очередь (задержка секунды — приемлема).

**4. Проблема знаменитости.** 50 млн подписчиков × пост = 50 млн записей — fan-out взрывается. **Гибрид**: для аккаунтов выше порога подписчиков посты НЕ раскладываются; при чтении лента = precomputed-список + on-the-fly подмешивание постов знаменитостей из их собственных таймлайнов (их мало — единицы на пользователя). Это стандартный ответ, который ждут.

**5. Хранение.** Посты — БД с шардированием по user_id автора (таймлайн автора — один шард) + кэш горячих постов. Граф подписок — отдельное хранилище с быстрым «кто подписан на X» (для fan-out) и «на кого подписан Y» (для pull-части). Ленты — Redis (потеря допустима: лента перестраивается pull-ом, это и есть fallback).

**6. Пагинация.** Не offset (лента живая — сдвиги и дубли), а **cursor** по (created_at, post_id): «дай посты старше курсора». Новые посты сверху — отдельным запросом «после курсора».

**7. Ранжирование.** Хронология — просто список; алгоритмическая лента = отдельный сервис скоринга поверх кандидатов (candidates from fan-out → score → rerank), важно назвать это как расширение, не закапываться.

**Trade-offs для проговаривания:** задержка появления поста vs стоимость чтения; консистентность ленты (eventual — норм); ограничение размера precomputed-ленты; идемпотентность fan-out-воркеров (дубли в очереди).
