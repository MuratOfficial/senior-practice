Спроектируйте rate limiter для API: N запросов в окно на пользователя/ключ, кластер из десятков инстансов API. Разберите алгоритмы (fixed window, sliding window, token bucket), хранение состояния и поведение при отказе лимитера.

<!-- answer -->

**1. Алгоритмы.**

- **Fixed window**: счётчик на окно (`user:минута → count`). Дёшево, но граничный всплеск: 100 в 11:59:59 + 100 в 12:00:01 = 200 за 2 секунды.
- **Sliding window log**: хранить timestamp каждого запроса, считать точное окно. Точен, но память O(запросов) — дорого на больших лимитах.
- **Sliding window counter**: интерполяция двух соседних fixed-окон (`prev*перекрытие + curr`) — компромисс, которого почти всегда достаточно.
- **Token bucket**: бакет ёмкости B пополняется со скоростью R токенов/сек; запрос забирает токен. Даёт **и** среднюю скорость, **и** контролируемые бёрсты (B). Обычно лучший дефолт (используется Stripe, AWS).
- **Leaky bucket** — выравнивание исходящего потока (очередь с постоянной скоростью), для API-лимитов реже.

**2. Распределённое состояние.** Инстансы API stateless → счётчики в **Redis**: ключ на (клиент, правило), атомарность через **Lua-скрипт** (прочитать-обновить-решить одной операцией; иначе race «оба прочитали 99, оба пропустили»). Token bucket в Redis: хранить `(tokens, last_refill_ts)`, при запросе Lua пересчитывает пополнение по прошедшему времени — не нужен фоновый процесс. TTL на ключах — самоочистка.

**3. Оптимизация: локальный + глобальный.** Поход в Redis на каждый запрос — +RTT и точка отказа. Варианты: локальный кэш с батч-синхронизацией (инстанс списывает квоту блоками, небольшая неточность допустима); либо лимитер в **API gateway** (Envoy/NGINX + Redis) — один слой, приложения не трогаем.

**4. Отказ лимитера.** Главный вопрос дизайна: Redis недоступен — что делаем? **Fail-open** (пропускать) — для защиты от злоупотреблений: лимитер не должен ронять весь API. **Fail-closed** — только если лимит защищает от реального вреда (платежи, дорогие операции). Ответить «зависит, и вот критерий» — то, что ждут.

**5. Ответ клиенту.** `429 Too Many Requests` + `Retry-After`, заголовки `X-RateLimit-Limit / -Remaining / -Reset` (или стандартизируемый `RateLimit-*`). Разные правила: per-user, per-IP, per-endpoint, глобальный SLA-лимит — комбинируются, срабатывает самый строгий.

**6. Расширения**: иерархические квоты (организация → ключ), динамическое снижение лимитов при перегрузке (load shedding), отдельные лимиты на дорогие операции (мутации vs чтения), distributed rate limiting между дата-центрами (обычно каждый DC лимитирует свою долю).
