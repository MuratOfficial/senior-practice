Зачем нужен AsyncLocalStorage и какую проблему он решает в однопоточном Node с конкурентными запросами? Как он работает поверх async_hooks? Приведите пример сквозного request-id и назовите ограничения.

<!-- answer -->

**Проблема.** Node однопоточный, но обрабатывает **много запросов конкурентно** — их асинхронные операции чередуются на одном event loop. Поэтому «глобальная» переменная-модуль вроде `currentUser` не работает: пока один запрос ждёт БД, обработается другой и перезапишет её. Прокидывать `ctx` параметром через все слои (логгер, репозиторий, сервис) — шумно и заразно.

**AsyncLocalStorage** даёт *логическое* thread-local хранилище: значение, привязанное к цепочке асинхронных вызовов. Внутри `run(store, cb)` все `await`, промисы, таймеры и колбэки, порождённые из `cb`, видят один и тот же `getStore()` — даже глубоко в стеке, без передачи аргументов. Под капотом — `async_hooks`: Node отслеживает создание/переключение асинхронных ресурсов и переносит контекст по цепочке.

```js
import { AsyncLocalStorage } from "node:async_hooks";
const als = new AsyncLocalStorage();

// middleware: заводим контекст на весь запрос
app.use((req, res, next) => {
  als.run({ requestId: crypto.randomUUID(), userId: req.user?.id }, next);
});

// где угодно глубоко внутри — без прокидывания
function log(msg) {
  const { requestId } = als.getStore() ?? {};
  console.log(`[${requestId}] ${msg}`);
}
```

Канонические применения: сквозной **request-id** для логов/трейсинга (так работают OpenTelemetry, pino, Sentry), текущий пользователь/тенант, транзакция БД, привязанная к запросу.

**Цена.** Раньше `async_hooks` заметно тормозил; в современных Node накладные расходы малы, но не нулевые — на сверхгорячих путях (десятки млн операций/с) их стоит измерять. `getStore()` дёшев; дорого — включение самого механизма отслеживания контекста.

**Ограничения и ловушки:**

- Контекст **не переживает** выход за пределы отслеживаемой цепочки: колбэк, привязанный к долгоживущему `setInterval`/`EventEmitter`, созданному *вне* `run`, увидит не тот store или `undefined`. Захватывайте нужные значения в замыкание заранее.
- Кэширование промиса, созданного в одном запросе, и его переиспользование в другом «протащит» чужой контекст — осторожно с межзапросными кэшами.
- Пулы/очереди, шедулящие работу вне исходного `run`, теряют контекст — оборачивайте задачу в `als.run` при постановке.
- Не замена явной передачи данных, где контракт важен: ALS — для сквозных, «инфраструктурных» данных (логи, трейс), а не для бизнес-аргументов функции.
