Сервис на Node.js постепенно съедает память и перезапускается по OOM. Как диагностировать утечку, какие источники утечек типичны и как ограничена память V8?

<!-- answer -->

**Модель памяти.** RSS процесса = V8 heap (объекты JS) + external/ArrayBuffers (Buffer живёт вне heap!) + стек и нативные структуры. Лимит старого поколения heap задаётся `--max-old-space-size` (по умолчанию порядка 2 ГБ — в контейнере с бóльшим лимитом надо поднимать явно, иначе OOM от V8 раньше лимита контейнера; с меньшим — наоборот, контейнер убьёт процесс раньше). Рост **heap** — утечка JS-объектов; рост **RSS при стабильном heap** — Buffers/нативные ресурсы/фрагментация.

**Типичные источники утечек:**

- **Кэш без границ**: глобальный `Map` «навсегда» — заменять на LRU с лимитом (lru-cache) или `WeakMap` (если ключ — объект с естественным временем жизни);
- **Слушатели событий**: подписка на долгоживущий emitter (socket, process) без `removeListener` — предупреждение MaxListenersExceededWarning это ловит;
- **Замыкания в долгоживущих структурах**: колбэк в очереди/таймере удерживает большой контекст запроса;
- **Незакрытые ресурсы**: соединения, стримы без завершения, `setInterval` без clear;
- **AsyncLocalStorage/контексты запросов**, попавшие в глобальные структуры.

**Диагностика:**

1. **Подтвердить утечку**: метрики `process.memoryUsage()` (heapUsed, rss, external) в мониторинг; рост после каждого GC — утечка, пила — норма.
2. **Heap snapshot**: два снимка с интервалом (нагрузка между ними) → сравнение в Chrome DevTools (Comparison): что выросло по количеству/размеру, смотреть **retainers** — цепочку, удерживающую объект от GC. Снять с прода: `node --inspect` + DevTools, программно `v8.writeHeapSnapshot()`, или по сигналу (`--heapsnapshot-signal=SIGUSR2`). Снимок останавливает event loop на секунды и весит с heap — на одной инстансе, выведенной из балансировки.
3. **Допрофилирование**: allocation timeline/sampling (`--heap-prof`) — откуда аллокации.

**Практика прода:** алерт на тренд heapUsed; рестарты по памяти — костыль, маскирующий проблему (но валидный ограничитель blast radius); нагрузочный тест с профилем — утечки видны за минуты, а не дни. Частая «неутечка»: недостаточный `--max-old-space-size` при легитимно большом рабочем наборе, либо external-память Buffers при работе с файлами/сокетами — это не чинится поиском утечек JS.
