TypeScript — структурно типизированный язык. Что это значит на практике? Почему код ниже ведёт себя по-разному, и как получить номинальную типизацию, когда она нужна?

```ts
interface Point { x: number; y: number }
declare function draw(p: Point): void;

draw({ x: 1, y: 2, z: 3 });        // ошибка
const p = { x: 1, y: 2, z: 3 };
draw(p);                            // ок
```

<!-- answer -->

**Структурная типизация:** совместимость определяется формой, не именем — если у значения есть все нужные поля нужных типов, оно подходит. `p` имеет x и y → совместим с Point, лишний `z` не мешает (ширина типа больше — это подтип).

**Excess property check** — исключение для **свежих объект-литералов**: литерал, передаваемый напрямую туда, где ожидается тип, проверяется строго на лишние поля. Логика: у литерала лишнее поле — почти наверняка опечатка (`colour` vs `color`), другого способа его использовать нет. Присвоенная переменная теряет «свежесть» — проверка не срабатывает. Это линт-эвристика компилятора, а не свойство системы типов.

**Следствия структурности:**

- два одинаковых по форме типа взаимозаменяемы, даже если семантически различны: `UserId` и `OrderId` как `string` перепутать легко;
- пустой интерфейс совместим со всем;
- приватные поля классов дают **квазиноминальность** — классы с private совместимы только с собой.

**Branded / nominal types** — стандартный обход:

```ts
type UserId = string & { readonly __brand: "UserId" };
const toUserId = (s: string) => s as UserId; // единственная точка входа

declare function getUser(id: UserId): User;
getUser("abc");          // ошибка — обычная строка
getUser(toUserId("abc")); // ок
```

Брэнд существует только в типах (в рантайме это строка). Применять для: идентификаторов, денег/валют, проверенных значений (`ValidatedEmail`, `SanitizedHtml`) — тип кодирует факт прохождения проверки.

**Вариантность:** параметры функций проверяются **контравариантно** при `strictFunctionTypes` (обработчик более общего типа подходит там, где ждут более узкий), возвращаемые значения — ковариантно. Методы интерфейсов исторически бивариантны — одна из причин предпочитать свойства-функции (`on: (e: Event) => void`) методам в строгих API.
