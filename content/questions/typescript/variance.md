Что такое ковариантность и контравариантность на примере функций и коллекций? Почему параметры функций контравариантны, а `strictFunctionTypes` не распространяется на методы? Где TS сознательно жертвует soundness ради удобства?

<!-- answer -->

**Вариантность** описывает, как отношение подтипов между `Dog <: Animal` переносится на конструкции над ними.

- **Ковариантность** — направление сохраняется: `Dog[] <: Animal[]`, `Promise<Dog> <: Promise<Animal>`. Так ведут себя **возвращаемые значения** и контейнеры-читатели.
- **Контравариантность** — направление переворачивается: для функций **параметры**. `(x: Animal) => void` является подтипом `(x: Dog) => void`, а не наоборот. Интуиция: обработчик, готовый принять любое `Animal`, безопасно подставить туда, где ждут обработчик `Dog` — он справится и с более общим входом. А узкий `(x: Dog) => void` нельзя дать туда, где могут передать «любое Animal»: он сломается на `Cat`.

```ts
type Handler<T> = (x: T) => void;
const animalH: Handler<Animal> = (a) => console.log(a.name);
const dogH: Handler<Dog> = (d) => d.bark();

let h: Handler<Dog> = animalH; // ok: параметр контравариантен
// let h2: Handler<Animal> = dogH; // ошибка при strictFunctionTypes
```

**`strictFunctionTypes`** включает эту строгую (контравариантную) проверку параметров для типов функций. Но с оговоркой: она **не применяется к методам**. Метод, объявленный синтаксисом `foo(x: T): void`, проверяется **бивариантно** (в обе стороны), а свойство-функция `foo: (x: T) => void` — контравариантно. Причина — прагматическая: без биваринтности методов ломались бы `Array<T>` и множество DOM/встроенных API (например `Array.prototype.forEach`), где это исторически удобно. Это сознательная дыра в soundness.

**Массивы ковариантны и это unsound:**

```ts
const dogs: Dog[] = [new Dog()];
const animals: Animal[] = dogs; // TS разрешает (ковариантность)
animals.push(new Cat());         // компилятор молчит — а в dogs теперь Cat!
```

Массив одновременно и читается (хочет ковариантности), и пишется (требует контравариантности) — строго он должен быть инвариантным, но TS выбрал ковариантность ради эргономики. `ReadonlyArray<T>` безопасен, потому что только читается.

**Маркеры `in`/`out`** (TS 4.7) позволяют явно аннотировать вариантность параметра дженерика: `interface Box<out T>` (ковариантный, только на выходе), `<in T>` (контравариантный, только на входе), `<in out T>` (инвариантный). Обычно TS выводит вариантность сам; маркеры нужны как документация намерения и для ускорения проверки в сложных рекурсивных типах.
