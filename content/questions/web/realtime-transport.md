Сравните способы доставки обновлений с сервера в браузер: short/long polling, Server-Sent Events, WebSocket. Критерии выбора, ограничения, устойчивость соединений и масштабирование.

<!-- answer -->

**Short polling** — запрос по таймеру. Просто, но latency = интервал, нагрузка впустую. Годится для редких некритичных обновлений.

**Long polling** — сервер держит запрос открытым до события/таймаута, клиент сразу переспрашивает. Latency почти realtime, работает через любые прокси; минусы: накладные на переустановку, порядок/пропуски при переподключении решать самому (курсор). Исторический fallback.

**SSE (EventSource)** — сервер держит HTTP-поток `text/event-stream`, шлёт события **только вниз**. Плюсы: обычный HTTP (проходит корпоративные прокси, мультиплексируется в HTTP/2), **автопереподключение встроено** + `Last-Event-ID` — докат пропущенного из коробки, просто на сервере. Минусы: однонаправленный (вверх — обычные POST), только текст; на HTTP/1.1 ест одно из 6 соединений на домен (на HTTP/2 не проблема). **Идеален для**: нотификации, лента, прогресс задач, стриминг LLM-токенов.

**WebSocket** — апгрейд HTTP до постоянного двунаправленного TCP-фрейминга. Плюсы: минимальная latency, бинарные данные, двусторонность (чат, игры, совместное редактирование, торговля). Минусы: свой протокол поверх (формат сообщений, ack, переподключение и докат — **всё руками**); stateful-соединения усложняют деплой (рестарт рвёт всех) и балансировку; некоторые прокси режут.

**Правило выбора:** «серверу нужно пушить, клиенту хватает HTTP для отправки» → **SSE** (недооценён и покрывает большинство продуктовых задач); настоящий двусторонний интерактив с высокой частотой → WebSocket; иначе — polling.

**Устойчивость (для обоих):** heartbeat (ping/pong или комментарии в SSE) — обнаружение мёртвых соединений (TCP сам не скажет); реконнект с **экспоненциальным backoff + jitter** (иначе thundering herd после рестарта сервера); **докат пропущенного** — события с id + запрос «с последнего id» (в SSE встроено, в WS — свой протокол); идемпотентность обработки на клиенте.

**Масштабирование WS:** балансировка L4/L7 (sticky не обязателен для чистого WS — соединение и так живёт на одном инстансе), состояние «кто где подключён» — во внешнем реестре, доставка между инстансами — pub/sub (Redis/NATS); graceful drain при деплое (перестать принимать, дождаться/мигрировать клиентов). Упомянуть WebTransport (HTTP/3) как развитие.
