Страница медленно загружается и «дёргается». По каким метрикам вы оцените проблему и какие меры примените — по этапам: доставка, критический путь рендера, JS, изображения?

<!-- answer -->

**Метрики (Core Web Vitals):** **LCP** (< 2.5 c) — отрисовка крупнейшего элемента, «когда видно главное»; **INP** (< 200 мс) — отзывчивость на взаимодействия за всю сессию; **CLS** (< 0.1) — визуальная стабильность. Диагностика: Lighthouse/DevTools Performance — лабораторно; CrUX/RUM — реальные пользователи (важнее).

**Доставка:** CDN ближе к пользователю; HTTP/2+ (мультиплексирование); сжатие Brotli; кэширование (immutable-ассеты с хэшем); ранние соединения — `preconnect` к критичным доменам.

**Критический путь рендера:** CSS блокирует рендер — минимизировать, критический CSS инлайном, остальное асинхронно; шрифты — `font-display: swap` + preload + подгонка fallback-метрик (иначе CLS/невидимый текст); `<script defer/module>` — парсинг HTML не блокируется; LCP-изображение — `fetchpriority="high"` и **не** лениво.

**JavaScript — обычно главный виновник:**

- **Меньше отгружать:** code splitting по маршрутам, динамический `import()` тяжёлых фич, tree shaking, анализ бандла (кто притащил 300 КБ библиотеку ради одной функции), RSC/островная архитектура — интерактивный JS только там, где интерактивность.
- **Меньше блокировать:** длинные задачи > 50 мс дробить (`scheduler.yield`), тяжёлое — в Web Worker; INP чаще всего портят синхронные обработчики + каскадные ререндеры.

**Изображения/медиа:** современные форматы (AVIF/WebP), `srcset/sizes` под вьюпорт, `loading="lazy"` вне первого экрана, явные `width/height` (резервируют место — CLS).

**CLS-источники:** изображения/эмбеды без размеров, контент, «впрыгивающий» сверху (баннеры), шрифтовой сдвиг, динамические вставки без резервирования места.

**Подход senior:** измерить → найти узкое место (не оптимизировать вслепую) → чинить самое дорогое → поставить бюджеты производительности в CI и RUM-мониторинг, чтобы не регрессировать.
