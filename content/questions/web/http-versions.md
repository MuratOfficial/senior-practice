Сравните HTTP/1.1, HTTP/2 и HTTP/3: соединения, мультиплексирование, head-of-line blocking. Как версии протокола влияли на практики фронтенда?

<!-- answer -->

**HTTP/1.1:** текстовый; одно соединение — один запрос за раз (keep-alive переиспользует, но последовательно; pipelining не взлетел). Браузеры открывали ~6 TCP-соединений на хост. **HOL-blocking на уровне HTTP**: медленный ответ держит очередь. Отсюда практики эпохи: бандлинг всего в один файл, CSS-спрайты, доменный шардинг, инлайн ресурсов.

**HTTP/2:** бинарный; **одно TCP-соединение, мультиплексирование** — параллельные потоки (streams) с чередованием кадров; сжатие заголовков HPACK; приоритеты. HTTP HOL-blocking решён. Практики изменились: шардинг вреден (ломает одно соединение), гранулярные файлы допустимы (лучше кэш-инвалидация), но **не** тысячи мелких (накладные на поток, сжатие хуже). Server Push — фактически умер, заменён ранними хинтами (103) и preload.

Осталась проблема: **TCP HOL-blocking** — потерянный пакет тормозит **все** потоки внутри соединения (TCP гарантирует порядок байтов, не зная о потоках HTTP/2). На плохих сетях HTTP/2 может быть хуже 1.1 с его шестью независимыми соединениями.

**HTTP/3 = HTTP поверх QUIC (UDP):**

- потоки независимы **на транспортном уровне** — потеря пакета тормозит только свой поток (решён и TCP HOL);
- TLS 1.3 встроен: рукопожатие 1-RTT, **0-RTT** для повторных соединений;
- **connection migration** — соединение переживает смену сети (Wi-Fi → LTE), так как идентифицируется connection ID, а не четвёркой адресов.

Выигрыш заметен на мобильных, нестабильных сетях и дальних RTT; в идеальной сети разница с HTTP/2 мала. Обнаружение — заголовок `Alt-Svc`/DNS HTTPS-записи.

**Практический вывод:** HTTPS обязателен (2/3 работают только с TLS); включается на CDN/балансировщике прозрачно для приложения; бэкенд-сервисы между собой часто остаются на HTTP/1.1/2 — и это нормально.
