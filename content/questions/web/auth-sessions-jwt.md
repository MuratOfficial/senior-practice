Сравните серверные сессии и JWT: компромиссы, отзыв, масштабирование. Где хранить токены в браузере? Когда JWT — неправильный выбор?

<!-- answer -->

**Серверные сессии:** после логина сервер создаёт запись (Redis/БД), клиенту — непрозрачный id в куке (`httpOnly, Secure, SameSite`). Каждый запрос — lookup сессии.

- ✅ мгновенный отзыв (удалил запись — разлогинен), «выйти со всех устройств», просто и понятно;
- ❌ хранилище сессий — общая зависимость всех инстансов (Redis это решает; «stateless-миф» о немасштабируемости сессий давно устарел).

**JWT:** подписанный (JWS) токен с claims (`sub`, `exp`, роли); сервер проверяет подпись — **без обращения к хранилищу**. Самодостаточность = главный компромисс:

- ✅ верификация локально: микросервисы/API-гейтвеи проверяют токен без общего state; кросс-доменные сценарии, делегирование (OAuth);
- ❌ **отзыв до истечения невозможен** без чёрного списка (а это снова state и вся «stateless»-выгода тает); данные в claims устаревают (роль поменяли — токен живёт); токен больше по размеру; криптоошибки конфигурации (принятие `alg: none`, путаница HS/RS) исторически били больно.

**Стандартный гибрид:** короткоживущий **access** (5–15 мин, JWT) + долгоживущий **refresh** (httpOnly-кука, хранится/отзывается на сервере, ротация с детекцией повторного использования). Компрометация access ограничена минутами; контроль — на refresh.

**Хранение в браузере:** localStorage — читается любым XSS, не рекомендуется для токенов. Куки httpOnly — недоступны JS (но CSRF-защита обязательна: SameSite + токен). Вариант «в памяти JS + refresh в httpOnly-куке» — частый компромисс SPA.

**Когда JWT не нужен:** классическое приложение один-бэкенд + браузер — сессии проще и безопаснее. JWT оправдан между сервисами, для API-клиентов, в федерации.

**OAuth 2.0 vs OIDC:** OAuth — протокол **делегирования доступа** («приложение X может читать мои файлы»), access token — для ресурса, не доказательство личности. **OIDC** — слой аутентификации поверх: добавляет `id_token` (JWT о том, *кто* вошёл) и userinfo. «Login with Google» — это OIDC. Для SPA/мобильных — Authorization Code + **PKCE** (implicit flow устарел).
