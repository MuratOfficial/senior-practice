Чем WeakMap/WeakSet отличаются от Map/Set и какие задачи решают? Что такое WeakRef и FinalizationRegistry, почему их использование не рекомендуют? Как слабые ссылки связаны со сборкой мусора?

<!-- answer -->

**Сильная ссылка** удерживает объект от GC. `Map`/`Set` держат ключи и значения сильно: положил объект ключом — он не соберётся, пока жива сама Map. Это скрытая утечка при кэшировании по объектам.

**WeakMap/WeakSet** держат ключи (только объекты, не примитивы) **слабо**: если на объект-ключ не осталось других ссылок, пара удаляется автоматически, память освобождается. Цена — приватность GC: коллекцию **нельзя перебрать**, нет `size`, `keys()`, `clear()`. Причина принципиальная: момент сборки недетерминирован, поэтому «сколько сейчас элементов» — вопрос без стабильного ответа, и итерация ломала бы детерминизм программы.

**Два канонических применения WeakMap:**

```js
// 1. Приватные данные, привязанные к экземпляру — уходят вместе с ним
const privates = new WeakMap();
class Api {
  constructor(token) { privates.set(this, { token }); }
  request() { return fetch("/x", { headers: { auth: privates.get(this).token } }); }
}

// 2. Мемоизация по объекту-аргументу без утечки
const cache = new WeakMap();
function heavy(obj) {
  if (cache.has(obj)) return cache.get(obj);
  const result = compute(obj);
  cache.set(obj, result); // когда obj умрёт, запись исчезнет сама
  return result;
}
```

Метаданные для DOM-узлов, «уже обработан» для объектов, декорирование чужих инстансов — всё это WeakMap, чтобы не мешать GC.

**WeakRef** — слабая ссылка на один объект: `ref.deref()` вернёт объект **или `undefined`**, если он уже собран. **FinalizationRegistry** — регистрация колбэка, который *может* вызваться после сборки объекта (для очистки внешних ресурсов).

**Почему не рекомендуют:** момент и сам факт вызова финализатора **не гарантированы** — GC волен не собирать объект вовсе, собрать поздно, а при выходе из процесса финализаторы могут не выполниться. Логику корректности на них строить нельзя. Легитимные ниши узкие: кэши, которые *можно* потерять (LRU поверх WeakRef), освобождение WASM/нативных хэндлов как страховка поверх явного `close()`. По умолчанию — детерминированное освобождение (`try/finally`, `AbortController`, явный dispose/`Symbol.dispose`), а слабые ссылки — крайний случай.
