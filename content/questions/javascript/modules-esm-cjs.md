В чём принципиальные отличия ES-модулей от CommonJS: разрешение, время загрузки, семантика экспортов, циклические зависимости? Как они взаимодействуют в Node.js?

<!-- answer -->

| | CommonJS | ESM |
|---|---|---|
| Загрузка | синхронная, в рантайме | статический разбор, асинхронная |
| `require`/`import` | обычная функция, можно в if | декларации хойстятся, путь — литерал (есть динамический `import()`) |
| Экспорт | копия значения на момент require | **живые связи** (live bindings) на переменные |
| this в модуле | `module.exports` | `undefined` |
| Tree shaking | плохо (динамичность) | да (статический граф) |

**Живые связи** — ключевое отличие: импортёр видит актуальное значение экспортированной переменной, а не снапшот. Поэтому ESM лучше переживает **циклические зависимости**: граф разбирается заранее, связи существуют до выполнения (хотя чтение до инициализации даст TDZ-ошибку). В CJS цикл возвращает **частично заполненный** `module.exports` — источник трудноуловимых багов.

**Node.js интероп:**

- Режим определяется полем `"type": "module"` в package.json или расширением `.mjs`/`.cjs`.
- Из ESM можно импортировать CJS (default = module.exports, именованные — эвристикой статического анализа).
- Из CJS **нельзя** синхронно `require` ESM (исторически; в новых версиях Node появился `require(esm)` для модулей без top-level await) — иначе только `await import()`.
- `__dirname`/`__filename` нет в ESM → `import.meta.url` + `fileURLToPath`.

**Dual package hazard:** пакет, поставляющий и CJS-, и ESM-сборку через `exports`, может быть загружен дважды — два экземпляра состояния (`instanceof` ломается). Решения: единая точка состояния, или один формат + обёртка.

**Top-level await** есть только в ESM: модуль становится асинхронным, и все его импортёры ждут его инициализации — злоупотребление задерживает старт всего графа.
