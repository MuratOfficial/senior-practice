Объясните, как устроен Event Loop в браузере. Что такое микрозадачи и макрозадачи, в каком порядке они выполняются и где в этом цикле рендеринг?

Что выведет этот код и почему?

```js
console.log("1");
setTimeout(() => console.log("2"), 0);
Promise.resolve().then(() => console.log("3"));
console.log("4");
```

<!-- answer -->

**Вывод: `1 4 3 2`.**

JS однопоточен: есть call stack, а Event Loop раз за разом берёт задачи из очередей, когда стек пуст.

**Порядок одной итерации цикла:**

1. Выполняется одна **макрозадача** (task): скрипт целиком, колбэк `setTimeout`/`setInterval`, обработчик события, I/O.
2. После неё выполняются **все** накопленные **микрозадачи** (microtasks): колбэки промисов (`.then`), `queueMicrotask`, `MutationObserver`. Если микрозадача добавляет новые микрозадачи — они выполняются в этой же фазе, до чего-либо ещё.
3. При необходимости браузер выполняет **рендеринг**: `requestAnimationFrame` → style → layout → paint (примерно раз в 16.6 мс при 60 FPS).

Поэтому в примере: синхронный код (`1`, `4`) → микрозадача промиса (`3`) → макрозадача таймера (`2`).

**Senior-нюансы:**

- Микрозадачи могут **заблокировать рендеринг навсегда** (бесконечная цепочка `then` не отдаёт управление), макрозадачи — нет: между ними браузер может отрисоваться.
- `setTimeout(fn, 0)` реально имеет минимальную задержку (клампинг ~4 мс при вложенности), и это только «не раньше чем», а не «ровно через».
- `requestAnimationFrame` привязан к кадру и вызывается **перед** paint — правильное место для анимаций; `setTimeout(fn, 16)` дрейфует и не синхронизирован с кадрами.
- Долгая задача (>50 мс) блокирует и ввод, и отрисовку — отсюда практики разбиения: `scheduler.yield()`, `setTimeout`-чанки, Web Workers.
