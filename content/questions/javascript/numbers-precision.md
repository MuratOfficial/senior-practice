Как устроены числа в JavaScript? Объясните, почему `0.1 + 0.2 !== 0.3`, что такое безопасные целые, когда нужен BigInt и как корректно работать с деньгами.

<!-- answer -->

**Один числовой тип.** `number` — это **double IEEE 754** (64 бита: 1 знак, 11 экспонента, 52 мантисса). Отдельного int нет: и `42`, и `0.5` — один тип.

**Почему `0.1 + 0.2 === 0.30000000000000004`:** 0.1 и 0.2 не представимы точно в двоичной системе (бесконечная двоичная дробь, как 1/3 в десятичной) — хранится ближайшее представимое число, ошибки складываются. Это свойство **всех** языков с IEEE 754 (Python, Java...), не «баг JS». Сравнение float: не `===`, а допуск — `Math.abs(a - b) < Number.EPSILON * scale`.

**Безопасные целые.** Мантисса 52 бита → целые точны только до **2⁵³ − 1** (`Number.MAX_SAFE_INTEGER` ≈ 9·10¹⁵). Дальше «дыры»: `2**53 + 1 === 2**53` — true. Практическое следствие: **64-битные id с бэкенда** (Twitter snowflake, БД bigint) нельзя парсить в number — теряются последние цифры. Передавать id строками (или JSON.parse с reviver/BigInt).

**BigInt** — целые произвольной длины: `123n`, `BigInt("9007199254740993")`. Нельзя смешивать с number в операциях (TypeError — явное приведение обязательно), нет поддержки в `JSON.stringify` из коробки, `Math.*` не работает. Использовать для: больших id, криптографии, точной целочисленной арифметики. Не для дробей.

**Деньги.** Никогда не float: хранить **в минимальных единицах целым числом** (копейки/центы) — в БД `bigint/integer`, в JS number (если суммы < 2⁵³) или BigInt/строки. Деление (раскладка скидки по позициям) — с явной стратегией распределения остатка. Альтернатива — decimal-библиотеки (big.js) на границах отображения; форматирование — `Intl.NumberFormat`.

**Ещё спрашивают:** `NaN` — единственное значение, не равное себе (`NaN !== NaN`; проверка `Number.isNaN`); `-0` существует (`Object.is(0, -0) === false`); `parseInt` vs `Number(...)` (`parseInt("12px") → 12`, `Number("12px") → NaN`); битовые операции работают на **int32** (обрезают до 32 бит — ловушка на больших числах).
