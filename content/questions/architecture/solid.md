Расшифруйте SOLID с примерами из веб-разработки (не абстрактные Animal/Dog). Какой принцип чаще всего нарушают и как принципы выглядят вне ООП?

<!-- answer -->

**S — Single Responsibility:** у модуля одна причина для изменения (один «заказчик» изменений). `UserService`, который валидирует, сохраняет, шлёт письмо и пишет метрики, меняется по четырём причинам → разделить: валидация (схема), репозиторий, нотификации, метрики (middleware). Самый нарушаемый принцип: «god-сервисы» и компоненты на 800 строк.

**O — Open/Closed:** расширяемость без правки существующего кода. Признак нарушения — растущий `switch` по типу в десяти местах. Решение — полиморфизм/реестр: новый способ оплаты = новый модуль, реализующий `PaymentProvider`, а не новая ветка в каждом switch.

**L — Liskov Substitution:** подтип обязан выполнять контракт базового типа — не только сигнатуры, но поведение (пред/постусловия, инварианты). Реальные нарушения: реализация интерфейса кэша, которая бросает на `delete` («read-only кэш»); метод, возвращающий null там, где базовый контракт обещает массив; мок, ведущий себя иначе, чем прод-реализация.

**I — Interface Segregation:** много узких интерфейсов вместо одного толстого — клиент не должен зависеть от методов, которые не использует. В TS это дёшево: функция принимает `Pick<Repo, "findById">` или узкий Protocol — легче тестировать и переиспользовать.

**D — Dependency Inversion:** модули высокого уровня зависят от **абстракций**, а не от конкретики; конкретика подставляется снаружи (DI). Бизнес-логика не импортирует `nodemailer` — она зависит от `Mailer`-интерфейса; SMTP/SES/фейк для тестов — детали сборки.

**Вне ООП:** те же идеи без классов — SRP: маленькие функции/модули и хуки с одной задачей; OCP: композиция, HOF, конфигурация поведением (стратегия = функция-параметр); DIP: передача зависимостей аргументами/контекстом вместо импорта синглтона.

**Мера:** SOLID — эвристики против стоимости изменений, не догма. Абстракция под «возможное будущее», которого нет (интерфейс с единственной реализацией навсегда, слои-прослойки), — сама по себе стоимость. Правило: вводить абстракцию на **второй** реализации или на границе, которую реально нужно мокать/менять.
