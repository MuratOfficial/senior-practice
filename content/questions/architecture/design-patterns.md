Какие паттерны проектирования вы реально применяете в веб-разработке? Приведите живые примеры из экосистемы JS/TS и объясните, какие классические паттерны стали идиомами языка.

<!-- answer -->

**Реально частые в вебе:**

- **Strategy** — взаимозаменяемые алгоритмы за одним интерфейсом: провайдеры оплаты/авторизации (passport strategies, Auth.js providers), способы доставки. В JS стратегия — просто функция-параметр.
- **Adapter** — чужой интерфейс → ваш: обёртки над SDK, `PrismaAdapter` для Auth.js, драйверы БД за общим интерфейсом.
- **Facade** — простой вход в сложную подсистему: сервисный слой над несколькими репозиториями и API; SDK-клиенты.
- **Observer / Pub-Sub** — реакция на события: EventEmitter, DOM events, подписки сторов (Zustand subscribe), RxJS. Отличие: Observer — подписчики знают субъект напрямую; Pub/Sub — через брокер/шину, стороны развязаны (Kafka, Redis pub/sub).
- **Decorator** — прозрачное добавление поведения: middleware (Express/Koa — цепочка декораторов над handler), HOC в React, Python `@decorator`, интерсепторы axios.
- **Factory / Builder** — конструирование: фабрики тестовых данных, `createClient(config)`, query builders (Knex — Builder в чистом виде).
- **Repository + Unit of Work** — изоляция доступа к данным от домена: тестируемость (подмена in-memory), возможность сменить хранилище, единая точка для кэша. При живом ORM полезен как **граница**: домен не знает про Prisma; UoW = транзакция вокруг набора изменений.
- **Circuit Breaker, Retry with backoff** — паттерны устойчивости для внешних вызовов (opossum).
- **CQRS** (разделение чтения/записи) — когда read-модель сильно отличается от write.

**Растворились в языке:** Singleton — модуль (кэш модулей ES) уже единственный экземпляр; Iterator — протокол итерации; Command — замыкание/функция первого класса; Strategy/Template Method — HOF; Proxy — встроенный `Proxy` (реактивность Vue/MobX на нём).

**Senior-фрейминг на интервью:** паттерн — общий словарь для решения, а не цель; называть паттерн стоит **после** того, как к нему привела задача (дублирование, тестируемость, связность), — код, написанный «под паттерн», обычно переусложнён.
