Сравните REST, GraphQL и gRPC: когда что выбирать? Разберите ключевые решения дизайна REST API: ресурсы, статусы, пагинация, ошибки, версионирование, идемпотентность.

<!-- answer -->

**Выбор протокола по потребителю:**

- **REST** — дефолт для публичных и внутренних CRUD-API: HTTP-семантика (кэширование, статусы, промежуточные прокси), простота, вездеходность.
- **GraphQL** — много разных клиентов с разными потребностями в данных (мобилка vs веб), агрегация нескольких источников, избавление от over/under-fetching. Цена: сложность сервера (N+1 → DataLoader, лимиты глубины/стоимости запросов), кэширование труднее.
- **gRPC** — межсервисное общение: бинарный Protobuf (компактно, быстро), строгие контракты с кодогенерацией, стриминг. В браузере — только через grpc-web/прокси.

**REST-решения:**

- **Ресурсы, не действия:** `POST /orders`, `GET /orders/{id}`; действия-исключения — подресурсом (`POST /orders/{id}/cancel`). Методы по семантике: PUT — полная замена (идемпотентен), PATCH — частичное, DELETE идемпотентен.
- **Статусы:** 400 — невалидный ввод, 401 — не аутентифицирован, 403 — нет прав, 404 — нет ресурса (или скрываем 403), 409 — конфликт состояния, 422 — семантика, 429 — rate limit. Ошибки — структурированный формат (RFC 9457 problem+json): code, message, details по полям, requestId.
- **Пагинация:** offset прост, но на больших смещениях медленный (`OFFSET 100000` читает всё) и «едет» при вставках; **cursor** (keyset: `WHERE (created_at, id) < ($1, $2) LIMIT n`) — стабилен и O(log n). Для UI со страницами offset допустим; для лент/синков — cursor.
- **Идемпотентность записи:** ретрай POST не должен создать два платежа → клиент шлёт `Idempotency-Key`, сервер хранит результат первого выполнения (уникальный индекс) и повторяет ответ.
- **Версионирование:** сначала — **эволюция без слома**: добавлять поля можно, удалять/переименовывать/менять типы — нельзя (клиенты должны игнорировать неизвестные поля). Слом неизбежен → `/v2` (явно, просто) или заголовок; поддерживать N и N−1, с deprecation-периодом и метриками использования старой версии.
- Также: единый стиль имён, фильтрация/сортировка (`?filter=&sort=`), rate limiting с заголовками, OpenAPI-спека как контракт (генерация клиентов, contract-тесты).
