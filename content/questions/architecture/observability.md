Из чего состоит observability распределённой системы? Разберите три столпа (логи, метрики, трейсы), их назначение и связывание, ключевые метрики сервиса и практику алертинга.

<!-- answer -->

**Три столпа и их роли:**

- **Метрики** — агрегированные числа во времени (Prometheus): дёшевы, хранятся долго, отвечают «**что** происходит» (RPS упал, латентность выросла) и питают алерты. Не отвечают «почему у конкретного запроса».
- **Логи** — события с контекстом: отвечают «**что именно** случилось» в конкретном случае. Обязательно **структурированные** (JSON: level, message, requestId, userId, поля) — иначе не ищутся; уровни осмысленно (error = требует действия); без секретов/PII.
- **Трейсы** — путь запроса через сервисы (OpenTelemetry): дерево спанов с таймингами — отвечают «**где** потратилось время / какой из 6 сервисов сбоит». Дороги в объёме → сэмплирование (в идеале tail-based: сохранять медленные/ошибочные).

**Связывание — главное:** сквозной **trace_id/correlation id** генерируется на входе, прокидывается через все вызовы (W3C traceparent) и пишется **в каждый лог**. Тогда путь диагностики: алерт по метрике → примеры трейсов с той же выборки → логи по trace_id. Без связывания три столпа — три несвязанных свалки.

**Какие метрики снимать.** Для каждого сервиса — **RED**: Rate (RPS), Errors (доля ошибок), Duration (латентность **перцентилями** — p50/p95/p99, среднее скрывает хвост; именно p99 чувствуют пользователи). Для ресурсов — **USE**: Utilization, Saturation (очереди!), Errors. Плюс бизнес-метрики (заказы/мин) — часто падают раньше технических.

**Алертинг:** алертить на **симптомы для пользователя** (SLO: доля ошибок, латентность), а не на причины (CPU 80% — не пейджить ночью); каждый алерт — actionable (есть runbook), иначе alert fatigue и проигнорированный настоящий инцидент. SLO/error budget: бюджет ошибок тратится → тормозим релизы, чиним надёжность.

**Практика внедрения:** OpenTelemetry SDK (vendor-neutral: авто-инструментация HTTP/БД + ручные спаны вокруг бизнес-операций); логи через контекст (AsyncLocalStorage в Node — requestId во все логи без протаскивания параметра); дашборд сервиса стандартный: RED + сатурация + зависимые сервисы.

**Observability vs мониторинг** (частый follow-up): мониторинг — заранее известные вопросы (дашборды, алерты на known failure modes); observability — способность задавать **новые** вопросы к системе без деплоя нового кода (высококардинальные данные, трейсы, structured events) — «почему медленно именно у пользователей из этого региона на этой версии клиента».
